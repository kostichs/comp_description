# План интеграции пайплайна поиска домашних страниц

## 1. Цель

Интегрировать функциональность из `test_serper.py` (поиск домашних страниц компаний) в основной рабочий пайплайн обработки информации о компаниях. Интеграция должна быть выполнена осторожно, поэтапно, с сохранением работоспособности существующего пайплайна и улучшением его возможностей по поиску веб-сайтов компаний.

## 2. Руководящие принципы

*   **Поэтапность**: Выполнять интеграцию шаг за шагом.
*   **Осторожность**: Минимизировать риск нарушения существующей функциональности.
*   **Тестирование**: Тщательно тестировать на каждом этапе.
*   **Модульность**: Рефакторить код из `test_serper.py` в виде переиспользуемого модуля.
*   **Консистентность**: Соблюдать стиль и структуру основного проекта.

## 3. Анализ и подготовка

### 3.1. Анализ `test_serper.py`

*   **Ключевые функции**:
    *   `normalize_name_for_domain_comparison`: Очистка имени компании.
    *   `check_domain_availability`, `check_domains_batch`: Асинхронная проверка доступности доменов.
    *   `find_domain_by_tld`: Поиск доменов по списку TLD.
    *   `parse_wikipedia_website`: Парсинг URL с Wikipedia.
    *   `get_wikidata_url`: Получение URL из Wikidata.
    *   `calculate_relevance_score`, `filter_wikipedia_links`: Оценка и фильтрация ссылок Wikipedia.
    *   `choose_best_wiki_link`: Выбор лучшей ссылки Wikipedia с помощью LLM (OpenAI).
    *   `extract_company_name_from_wiki_url`: Извлечение имени компании из URL Wikipedia.
    *   `process_company`: Оркестратор, выполняющий поиск через Wikidata, TLD, Google Serper, и выбирающий лучший результат.
*   **Зависимости**: `requests`, `python-dotenv`, `pandas`, `aiohttp`, `beautifulsoup4`, `openai`.
*   **Вход**: Имя компании.
*   **Выход**: Печать URL в консоль и возврат строки-статуса (`"wikidata"`, `"domains"`, etc.). Это нужно будет изменить.

### 3.2. Анализ основного пайплайна (на основе `README.md` и структуры проекта)

*   **Основной скрипт обработки**: Вероятно, `src/pipeline.py`.
*   **Точка входа для обработки компании**: Необходимо найти функцию или класс в `src/pipeline.py`, где обрабатывается каждая компания и где происходит текущий поиск веб-сайта (если он есть).
*   **Конфигурация**: Используется `.env` для API ключей (`OPENAI_API_KEY`, `SCRAPINGBEE_API_KEY`). `SERPER_API_KEY` нужно будет добавить или убедиться, что он обрабатывается аналогично.
*   **Выходные данные**: CSV-файл с общей информацией (`description`) и детальные JSON-файлы для каждой компании. Новый URL домашней страницы должен быть включен в эти выходные данные.
*   **Существующий поиск веб-сайта**: `README.md` упоминает "веб-скрейпинг, Wikipedia". Новый механизм должен либо заменить, либо дополнить существующий.

## 4. Рефакторинг `test_serper.py` в модуль

### 4.1. Создание нового файла модуля
    *   Создать файл `src/homepage_finder.py`.

### 4.2. Перенос и адаптация функций
    *   Скопировать все вспомогательные функции (`normalize_name_for_domain_comparison`, `check_domain_availability`, `check_domains_batch`, `find_domain_by_tld`, `parse_wikipedia_website`, `get_wikidata_url`, `calculate_relevance_score`, `filter_wikipedia_links`, `choose_best_wiki_link`, `extract_company_name_from_wiki_url`) из `test_serper.py` в `src/homepage_finder.py`.
    *   Адаптировать функцию `process_company`:
        *   Переименовать ее, например, в `find_company_homepage`.
        *   Она должна принимать `company_name: str` и, возможно, `serper_api_key: str`, `openai_api_key: str` (или полагаться на глобально загруженные ключи из `.env`).
        *   **Важно**: Изменить возвращаемое значение. Вместо строки статуса, функция должна возвращать найденный URL (строку) или `None`, если URL не найден. Опционально, можно возвращать словарь `{"url": "...", "source": "..."}` для дополнительной информации.
        *   Убрать из нее загрузку компаний из файла и `asyncio.run(main())` – модуль должен предоставлять функции, а не быть исполняемым скриптом.
    *   Убедиться, что все импорты корректны внутри нового модуля.

### 4.3. Обработка API ключей
    *   Функции, использующие API ключи (`choose_best_wiki_link` для OpenAI, основная логика `find_company_homepage` для Serper) должны получать их либо как аргументы, либо ожидать, что они загружены через `os.getenv()` (что предпочтительнее, если основной проект уже использует `.env`).

## 5. Интеграция `homepage_finder.py` в основной пайплайн

### 5.1. Определение точки интеграции

### 5.2. Импорт и вызов новой функции
    *   Импортировать `find_company_homepage` из `src.homepage_finder`.
    *   Вызвать `await find_company_homepage(company_name, serper_api_key=os.getenv("SERPER_API_KEY"), openai_api_key=os.getenv("OPENAI_API_KEY"))` (если `find_company_homepage` сделана асинхронной, что рекомендуется, учитывая `aiohttp`). Если она синхронная, то без `await`.

### 5.3. Использование результата
    *   Полученный URL от `find_company_homepage` использовать для дальнейших шагов (например, скрейпинг, если он предусмотрен) и для сохранения в результатах.
    *   **Стратегия**:
        *   Если `find_company_homepage` возвращает URL, использовать его как основной.
        *   Если существующий пайплайн имеет свой метод поиска URL, решить:
            *   Новый метод заменяет старый.
            *   Новый метод является основным, старый – как fallback.
            *   Оба метода используются, и выбирается лучший результат (требует логики выбора).
        *   Для начала, можно попробовать использовать новый метод как основной источник URL.

### 5.4. Обновление выходных данных
